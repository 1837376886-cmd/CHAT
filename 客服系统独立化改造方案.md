# 客服系统独立化改造方案

## 一、目标

将当前客服聊天模块改造为**独立部署的若依应用**：
- 客服业务数据（`cs_*`、`chat_*`、`chat_visitor` 等）使用**本地主库**
- 用户、角色、菜单、部门等系统数据从**另一套若依系统的数据库（从库）读取**
- Token 不共享，客服系统拥有独立的登录态
- 登录时只允许客服 / 管理员 / 开发角色进入

> 前提：另一套系统同样基于若依架构，表结构完全兼容。

---

## 二、整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                      客服系统（独立部署）                      │
│  ┌──────────────┐    ┌──────────────────────────────────┐   │
│  │   本地主库    │    │           从库（另一套若依）        │   │
│  │  cs_session  │    │  sys_user / sys_role / sys_menu  │   │
│  │  cs_message  │    │  sys_dept / sys_user_role ...    │   │
│  │  chat_visitor│    └──────────────────────────────────┘   │
│  │  cs_config   │                                          │
│  └──────────────┘                                          │
│                                                             │
│  Redis（独立）：Token、在线状态、未读数、排队队列                │
│  WebSocket（独立）：Netty WS 端口 9999                       │
└─────────────────────────────────────────────────────────────┘
```

## 三、数据源配置改造

### 3.1 引入动态数据源依赖

在 `ruoyi-framework/pom.xml` 中新增（若未引入）：

```xml
<dependency>
    <groupId>com.baomidou</groupId>
    <artifactId>dynamic-datasource-spring-boot-starter</artifactId>
    <version>3.6.1</version>
</dependency>
```

### 3.2 修改 application-druid.yml

```yaml
spring:
  datasource:
    type: com.alibaba.druid.pool.DruidDataSource
    dynamic:
      primary: master
      strict: false
      datasource:
        # 本地主库：客服业务表
        master:
          driver-class-name: com.mysql.cj.jdbc.Driver
          url: jdbc:mysql://localhost:3306/ruoyi-cs?useUnicode=true&characterEncoding=utf8&zeroDateTimeBehavior=convertToNull&useSSL=true&serverTimezone=GMT%2B8
          username: root
          password: xxxx
        # 从库：另一套若依的系统表
        slave:
          driver-class-name: com.mysql.cj.jdbc.Driver
          url: jdbc:mysql://另一套系统IP:3306/ruoyi-admin?useUnicode=true&characterEncoding=utf8&zeroDateTimeBehavior=convertToNull&useSSL=true&serverTimezone=GMT%2B8
          username: root
          password: xxxx
```

> 注意：若依原生的 `DruidConfig` 和 `MyBatisConfig` 需要适配，或直接使用动态数据源提供的自动配置，移除原生的单数据源 `MyBatisConfig` 中手动创建 `DataSource` 的逻辑。

### 3.3 数据源路由策略

| 包路径 / 模块 | 目标数据源 | 说明 |
|-------------|----------|------|
| `com.ruoyi.chat.mapper.*` | `master` | 客服业务 Mapper 走本地库 |
| `com.ruoyi.system.mapper.*` | `slave` | 系统表 Mapper 走从库 |
| `com.ruoyi.generator.mapper.*` | `slave` | 代码生成走从库（如保留） |
| `com.ruoyi.quartz.mapper.*` | `slave` | 定时任务走从库（如保留） |

实现方式（推荐二选一）：

**方式 A：@DS 注解（精确控制）**
在 `ruoyi-system` 的 Service 层或 Mapper 层批量加注解：

```java
@DS("slave")
@Service
public class SysUserServiceImpl implements ISysUserService { ... }
```

**方式 B：包级 AOP 拦截（更简洁）**
新增一个 `DataSourceAspect`：

```java
@Aspect
@Component
public class DataSourceAspect {
    @Before("execution(* com.ruoyi.system..*.*(..))")
    public void setSlaveDataSource() {
        DynamicDataSourceContextHolder.push("slave");
    }

    @After("execution(* com.ruoyi.system..*.*(..))")
    public void clearDataSource() {
        DynamicDataSourceContextHolder.poll();
    }
}
```

## 四、登录与角色校验改造

### 4.1 登录流程

保留客服系统自己的登录页面，独立生成 Token，Redis 也独立。

```
用户输入账号密码
    ↓
SysLoginService.login() → 查 slave 库 sys_user 验证密码
    ↓
生成 Token → 写入本地 Redis
    ↓
【新增】校验角色：仅客服/管理员/开发允许登录
    ↓
返回 Token
```

### 4.2 角色校验实现

在 `ruoyi-framework/.../web/service/SysLoginService.java` 的 `login()` 方法末尾，生成 Token 之前插入：

```java
// 登录成功后，校验用户角色
Set<String> roles = permissionService.getRolePermission(loginUser.getUser());
boolean isAllowed = roles.stream().anyMatch(r ->
    r.equals("customer_service") ||  // 客服
    r.equals("admin") ||             // 管理员
    r.equals("developer")            // 开发
);
if (!isAllowed) {
    throw new ServiceException("无权登录客服系统，请联系管理员分配客服角色");
}
```

> 角色标识符（`customer_service`）需与另一套若依中配置的角色编码保持一致。

### 4.3 菜单路由

`SysPermissionService.getMenuPermission()`、`SysMenuService.selectMenuTreeByUserId()` 等方法查的是 `sys_menu` 表，因为 `ruoyi-system` 已路由到 slave 库，所以**无需改动业务代码**，前端 `getRouters` 接口自动返回另一套系统配置的菜单。

**注意**：另一套若依的 `sys_menu` 中必须提前配置好客服菜单（客服工作台、客服人员设置、我的客服历史等）。

## 五、需要改造的文件清单

### 后端

| 文件 | 改造内容 |
|------|---------|
| `ruoyi-framework/pom.xml` | 引入 `dynamic-datasource-spring-boot-starter` |
| `ruoyi-admin/src/main/resources/application-druid.yml` | 改为 `spring.datasource.dynamic` 多数据源配置 |
| `ruoyi-framework/.../config/MyBatisConfig.java` | 移除或注释掉手动创建 `DataSource` 和 `SqlSessionFactory` 的单数据源逻辑（让动态数据源接管） |
| `ruoyi-framework/.../config/DruidConfig.java` | 适配多数据源，或移除（动态数据源自带 Druid 支持） |
| `ruoyi-framework/.../web/service/SysLoginService.java` | 登录成功后增加角色校验 |
| `ruoyi-system/.../service/impl/*ServiceImpl.java` | 批量添加 `@DS("slave")`，或使用包级 AOP 统一拦截 |
| 新增 `DataSourceAspect.java` | 包级数据源路由切面（如采用方式 B） |

### 前端

前端**无需改造**。`ruoyi-ui` 的登录、路由获取、权限判断逻辑完全不变，因为后端接口返回的数据结构仍然是标准若依格式。

## 六、边界情况处理

### 6.1 用户信息缓存

若依的 `TokenService` 会把 `LoginUser` 缓存到 Redis（`login_tokens:{uuid}`）。由于 `sys_user` 在从库，如果另一套系统修改了用户密码或状态，客服系统缓存里的用户信息会过期不一致。

**建议**：
- 缩短 Token 过期时间（如 30 分钟）
- 或在客服系统的用户管理页面提供“刷新缓存”功能
- 最彻底：客服系统不保留 `sys_user` 写权限，用户修改全部走另一套系统，客服系统只读

### 6.2 跨库事务

客服业务操作（如发送消息、创建会话）全部在 master 库，一般不会同时操作 slave 库。如果后续出现“写系统日志到 slave 库”的需求，需注意 `@Transactional` 默认只能管单数据源。跨库事务需要借助 Seata 或业务层补偿，**当前阶段可暂不考虑**。

### 6.3 代码生成与定时任务

`ruoyi-generator` 和 `ruoyi-quartz` 如果保留在客服系统，它们的表也在 slave 库，需确保这些模块的 Mapper 同样路由到 slave。

如果客服系统**不需要**代码生成和定时任务功能，可以直接：
- 从 `pom.xml` 移除 `ruoyi-generator` 和 `ruoyi-quartz` 依赖
- 前端菜单里隐藏对应入口

### 6.4 客服人员配置的用户筛选

当前客服系统的"客服人员设置"页面，新增客服时前端调用的是若依原生的 `listUser` 接口（查询全部用户）。独立化后，由于用户数据在从库维护，且并非所有用户都能当客服，需要调整该页面逻辑：

**问题**：
- 全部用户列表可能很大，前端一次性拉取性能差
- 普通用户（无客服角色）不应出现在新增客服的候选列表中

**建议改造**：
1. 后端新增接口（如 `/cs/staff/candidates`），查询具有 `customer_service` / `admin` / `developer` 角色且未设为客服的用户
2. 前端"客服人员设置"页面：
   - 列表页直接调用 `/cs/staff/list`（已是客服的用户）
   - 新增弹窗调用 `/cs/staff/candidates`（有资格但未设为客服的用户）

**涉及文件**：
- 后端：`CustomerServiceController` 新增接口、`CsStaffMapper` 自定义查询
- 前端：`ruoyi-ui/src/views/cs/staff/index.vue`、`ruoyi-ui/src/api/cs.js`

## 七、系统功能精简

客服系统独立后，应是一个**纯粹的客服业务应用**，不需要系统管理、代码生成、定时任务等非客服功能。需要从前后端两个层面做精简，避免给用户展示无关菜单，也避免误操作从库的系统数据。

### 7.1 后端模块移除

直接从 `ruoyi-admin/pom.xml` 中移除以下模块依赖：

```xml
<!-- 代码生成 -->
<dependency>
    <groupId>com.ruoyi</groupId>
    <artifactId>ruoyi-generator</artifactId>
    <version>${ruoyi.version}</version>
</dependency>

<!-- 定时任务 -->
<dependency>
    <groupId>com.ruoyi</groupId>
    <artifactId>ruoyi-quartz</artifactId>
    <version>${ruoyi.version}</version>
</dependency>
```

移除后重新编译，确认启动无误。

### 7.2 后端管理接口管控

`ruoyi-system` 模块中除了认证查询类接口（如 `SysLoginService`、`SysPermissionService`、`SysUserService.selectUserByUserName` 等），还包含大量**系统管理类接口**：
- 用户管理：新增、修改、删除用户
- 角色管理：新增、修改、删除角色
- 菜单管理：新增、修改、删除菜单
- 部门管理、岗位管理、字典管理、参数管理、通知公告...

**风险**：这些接口如果暴露在前端，操作时会直接修改从库（另一套若依的数据），造成数据不一致。

**建议处理方式（三选一）**：

| 方案 | 做法 | 优缺点 |
|------|------|--------|
| **A. 前端屏蔽（推荐）** | 保留后端接口，但前端不展示这些菜单，用户无法访问 | 改动最小，但接口理论上仍可通过工具调用 |
| **B. 接口层拦截** | 在 `SysUserController`、`SysRoleController` 等管理类 Controller 上加统一拦截，返回 `"系统数据请在管理后台维护"` | 安全，但需要改多处 Controller |
| **C. 直接移除** | 从客服源码中删除 `ruoyi-system` 里所有管理类 Controller/Service/Mapper/XML，只保留认证和查询相关代码 | 最干净，但后期若依升级时需手动同步认证相关代码 |

> **推荐方案 A + B 结合**：前端不展示，后端管理接口统一返回提示，双重保险。

### 7.3 前端页面精简

`ruoyi-ui/src/views/` 下与客服无关的目录：

| 目录 | 功能 | 处理方式 |
|------|------|----------|
| `system/` | 系统管理（用户、角色、菜单、部门、字典等） | **删除或移出项目**，因为这些页面操作的是从库系统表 |
| `tool/` | 代码生成 | **删除**，模块已移除 |
| `monitor/` | 系统监控、定时任务、在线用户等 | **删除**，客服系统不需要 |
| `dashboard/` | 首页数据面板 | 可保留但精简，或直接进入客服工作台 |

**精简后前端目录**：

```
src/views/
  login.vue          # 登录页
  cs/                # 客服业务（工作台、设置、历史）
  chat/              # 聊天室（如保留）
  dashboard/         # 精简版首页（可选）
  error/             # 错误页面
```

### 7.4 菜单与首页改造

**动态菜单**：
- 由于菜单数据来自另一套若依的 `sys_menu`，需确保另一套系统给客服角色只分配客服相关菜单。
- 如果另一套系统给客服角色分配了系统管理菜单，前端路由守卫可做二次过滤，只渲染 `path` 以 `/cs/` 或 `/chat/` 开头的路由。

**首页改造**：
- 若依默认登录后跳转到 `dashboard` 首页。建议改为登录后直接跳转到 **客服工作台**（`/system/chat/workbench` 或实际工作台路径）。
- 修改 `ruoyi-ui/src/store/modules/permission.js` 或登录成功后的跳转逻辑：

```javascript
// login.vue 登录成功后
this.$router.push('/system/chat/workbench')
```

**顶部导航栏**：
- 当前若依顶栏有"首页"、"系统管理"等入口，精简后只保留"客服工作台"、"我的客服历史"、"个人中心"、"退出登录"即可。

## 八、实施步骤（推荐顺序）

1. **环境准备**
   - 确认另一套若依数据库可被客服系统服务器访问
   - 在另一套若依中为客服账号分配角色和菜单（仅客服相关菜单）

2. **引入多数据源**
   - 添加动态数据源依赖
   - 修改 `application-druid.yml`
   - 移除/注释原生单数据源配置

3. **路由配置**
   - 给 `ruoyi-system` 模块加 `@DS("slave")` 或写 `DataSourceAspect`
   - 验证 `sys_user` 查询是否走了从库

4. **登录改造**
   - 在 `SysLoginService.login()` 中加入角色校验（仅客服/管理员/开发可登录）
   - 测试普通用户无法登录，客服角色可以登录

5. **功能验证**
   - 登录后菜单是否正常加载（应只有客服菜单）
   - 客服工作台会话、消息、WebSocket 是否正常
   - 权限注解是否正常

6. **系统精简**
   - 后端：从 `pom.xml` 移除 `ruoyi-generator`、`ruoyi-quartz`
   - 后端：给系统管理类 Controller 加统一拦截（防误改从库）
   - 前端：删除 `views/system`、`views/tool`、`views/monitor` 目录
   - 前端：登录成功后直接跳转客服工作台

7. **验证与收尾**
   - 确认系统管理菜单不可见、不可访问
   - 确认客服业务全流程正常
   - 打包部署测试

```

ruoyi-framework/      # 框架配置：多数据源、登录校验改造

ruoyi-ui/             # 前端：不变，可屏蔽系统管理菜单
ruoyi-generator/      # 可选：移除或保留（查 slave）
ruoyi-quartz/         # 可选：移除或保留（查 slave）
```

## 九、总结

| 维度 | 结论 |
|------|------|
| 技术可行性 | ✅ 完全可行，两套若依表结构天然兼容 |
| 代码改动量 | 中偏小，主要是配置层和登录校验 |
| 业务代码侵入 | 几乎为零，`ruoyi-chat` 一行业务代码不用改 |
| 部署独立性 | 高，客服系统可完全独立部署、独立迭代 |
| 后期维护成本 | 低，系统表由另一套若依统一维护，客服只关注业务 |
